最近都在用 claude code 接 glm-5 来 vibe coding，有了 ai 确实解决问题和写需求的速度突飞猛进，而且睡眠质量不高导致经常漏处理边界情况的 bug 少了很多。现在回想起来还是很感谢阮一峰推荐，在过年的时候买了智谱的 Pro 版 Coding Plan，后面纷纷涨价了，现在只希望当时能直接上 Max。搭配 claude code 真的很爽，当然 codex 也很爽，但是最近养 openclaw 花的比较快。。反而工作没用上 codex 了

有了 vibe coding 后，反而觉得以前的软件工程最佳实践更有价值了。比如 TDD，先写测试再写代码，以前时间充裕会写个 test 覆盖一下 happy path，忙起来的时候就直接写代码然后手动测了。现在 ai 写得飞快，看都看不过来，只能先让他写测试再写代码，自己闭环来搞，搞完我再来看，效率反而更高了。又比如 SDD(spec 驱动开发），在前面就确认好，后面就不用看他写错代码再来返工，常常一次能成型，真的很棒

自从习惯了命令行后，除非为了 opus4.6 这种顶级模型，不然真的用不惯 IDE 了，IDE 的速度明显比 CLI 的慢，而且很难闭环，受工作目录的约束也更大。命令行的缺点就是语音输入法用不了，现在闪电说和 CapsWriter 这种语音输入，都是利用剪贴板再贴进文本框的，windows terminal 下就弄不了。参考 [ 如何有效地给 10 个 Claude Code 打工 ](https://zhuanlan.zhihu.com/p/2007147036185744607) vibe coding 了一个，不过自己用不下去。。。总觉得非交互模式下，claude 结束的特别快，而且看不到 git diff，有点慌 …… 可能还是没进步到老板的视角 XD

AI 给的反馈确实很强，现在有了 ai，容易陷入一种我和成功只差一个 prompt 的错觉，一直肝一直肝，睡眠不足导致犯错误。还容易逛 X 后一直处于 FOMO（Fear Of Missing Out）的恐惧中，尝试一直优化自己的工作流。但是 AI 的进步又特别快，6 个月前的工具和方法论，也不一定适合当下的模型 Orz 反过来作为开发者，现在我觉得提供快速的有效反馈，才是自己的最大价值。这是同时对 AI 实习生和资深人类同事都有用的技能呢，搭配上合适的技术写作能力，能更好的推动组织发展
