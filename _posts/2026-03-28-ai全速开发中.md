最近用 Claude Code 接 GLM-5 来 vibe coding，有了 AI 加持，解决问题和写需求的速度突飞猛进。更关键的是，以前睡眠不足时经常遗漏边界情况导致的 bug，现在少了很多

回想起来很感谢阮一峰的推荐，趁着过年买了智谱的 Pro 版 Coding Plan，后来纷纷涨价，现在只后悔当时没直接上 Max。搭配 Claude Code 真的很爽，当然 Codex 也很香，不过最近养 OpenClaw 花得有点快……反而工作里没怎么用上 Codex 了

有了 vibe coding 后，反而觉得以前的软件工程最佳实践更有价值了

比如 TDD（测试驱动开发），先写测试再写代码。以前时间充裕会写个测试覆盖 happy path，忙起来就直接写代码手动测。现在 AI 写得飞快，看都看不过来，只能让它先写测试再写代码，自己形成闭环，完成后再来 review，效率反而更高

又比如 SDD（Spec 驱动开发），先把需求确认清楚，后面就不用看它写错代码再返工，常常一次就能成型，真的很棒

自从习惯了命令行后，除非为了 Opus 4.6 这种顶级模型，否则真的用不惯 IDE 了。IDE 的速度明显比 CLI 慢，而且很难形成闭环，受工作目录的约束也更大

命令行的缺点就是语音输入法用不了。现在闪电说、CapsWriter 这类语音输入工具，都是利用剪贴板再贴进文本框的，Windows Terminal 下就没法弄。参考[《如何有效地给 10 个 Claude Code 打工》](https://zhuanlan.zhihu.com/p/2007147036185744607) vibe coding 了一个，不过自己用不下去……总觉得非交互模式下 Claude 结束得特别快，而且看不到 git diff，心里有点慌……可能还是没进化到老板的视角 XD

AI 给的反馈确实很强，但也容易陷入一种「我和成功只差一个 prompt」的错觉，一直肝一直肝，睡眠不足反而容易犯错。还容易刷完 X 后一直处于 FOMO（Fear Of Missing Out）的焦虑中，不断尝试优化自己的工作流。但 AI 进步太快了，6 个月前的工具和方法论，不一定适合当下的模型 Orz

反过来想，作为开发者，现在我觉得**提供快速有效的反馈**，才是自己的最大价值。这是同时对 AI 实习生和资深人类同事都通用的技能，搭配上合适的技术写作能力，能更好地推动组织发展
